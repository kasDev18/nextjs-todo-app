# Taskflow

Taskflow is a production-oriented Next.js task board for authenticated users. It supports email/password sign-up with email verification, Kanban-style task management, due-date reminders, and PostgreSQL-backed persistence.

## Overview

The app is built with:

- `Next.js 16` App Router and `React 19`
- `TypeScript`
- `PostgreSQL`
- `Drizzle ORM`
- `Better Auth`
- `Nodemailer`
- `Tailwind CSS v4`, Radix/shadcn-style UI, `react-hook-form`, and `zod`

Core capabilities:

- Sign up, verify email, and sign in
- Create, edit, delete, and move tasks across `To Do`, `In Progress`, and `Done`
- Automatic project-code generation per task
- Due-date notifications for nearly expired and overdue tasks
- Reminder emails generated by a cron-based monitor
- Dockerized production build with health checks

## Installation Instructions

### Prerequisites

- `Node.js 22+`
- `pnpm 10+`
- `PostgreSQL 16+`
- An SMTP provider for verification and reminder emails
- Optional: `Docker` and `Docker Compose`

### Install dependencies

```bash
pnpm install
```

## Setup Guide

### 1. Create the environment file

```bash
cp .env.example .env
```

Most values in `.env.example` are already populated for local testing, so you can usually copy it directly and adjust only the environment-specific fields you need.

Update the values in `.env` before running the app.

Generate a `BETTER_AUTH_SECRET` before starting the app. You can use the Better Auth CLI:

```bash
pnpm exec @better-auth/cli secret
```

Or generate one with OpenSSL:

```bash
openssl rand -base64 32
```

Copy the generated value into `BETTER_AUTH_SECRET` in `.env`.

Important:

- The bundled values are suitable for testing, not production.
- Replace `BETTER_AUTH_SECRET` with a long random secret.
- Replace all `SMTP_*` values with your own mail server credentials.
- Keep `BASE_URL` and `NEXT_PUBLIC_BASE_URL` aligned with the URL users will access.
- Make sure the database name used in `DATABASE_URL`, `DOCKER_DATABASE_URL`, and `POSTGRES_DB` matches your actual database.

### 2. Required environment variables

| Variable | Required | Description |
| --- | --- | --- |
| `NODE_ENV` | Yes | `development` or `production` |
| `PORT` | Yes | App port, default `3000` |
| `BASE_URL` | Yes | Canonical server URL used by auth and reminder links |
| `NEXT_PUBLIC_BASE_URL` | Yes | Public app URL for client-side usage |
| `DATABASE_URL` | Yes | PostgreSQL connection string used by the app |
| `DATABASE_SSL` | Yes | `false`, `require`, or `verify-full` |
| `BETTER_AUTH_SECRET` | Yes | Better Auth signing secret |
| `SMTP_HOST` | Yes for email flows | SMTP host |
| `SMTP_PORT` | Yes for email flows | SMTP port |
| `SMTP_SECURE` | Yes for email flows | `true` or `false` |
| `SMTP_USER` | Yes for email flows | SMTP username |
| `SMTP_PASS` | Yes for email flows | SMTP password |
| `SMTP_FROM_NAME` | Yes for email flows | Sender display name |
| `SMTP_FROM_EMAIL` | Yes for email flows | Sender email address |
| `SEED_USER_EMAIL` | Optional | Existing account email to receive seeded tasks |
| `SEED_USER_NAME` | Optional | Display name used only if the seed script creates a user |
| `POSTGRES_DB` | Docker/local helper | Database name for containerized Postgres |
| `POSTGRES_USER` | Docker/local helper | Database user |
| `POSTGRES_PASSWORD` | Docker/local helper | Database password |
| `DOCKER_DATABASE_URL` | Optional | Override app database URL inside Docker Compose |
| `DOCKER_DATABASE_SSL` | Optional | Docker-specific SSL override |

### 3. Initialize the database for local development

This repository stores SQL migrations in `drizzle/`, but it does not currently expose a dedicated migration script in `package.json`. For a local Postgres instance, apply the SQL files in order before starting the app:

```bash
for file in drizzle/*.sql; do
  psql "$DATABASE_URL" -f "$file"
done
```

### 4. Start the app

```bash
pnpm dev
```

Open [http://localhost:3000](http://localhost:3000).

### 5. Create your first account

Visit `/signup`, register a user, and complete email verification before attempting to sign in. The app requires verified email/password auth.

Important:

- SMTP must be configured for verification emails to be delivered.
- If SMTP is not configured correctly, account creation may appear to succeed but the user will not be able to complete verification and sign in.

### 6. Seed sample task data

After you have a real account, seed tasks for that user:

```bash
pnpm db:seed
```

The seed script creates sample tasks and reminder rows for `SEED_USER_EMAIL`.

Important:

- Prefer setting `SEED_USER_EMAIL` to an already registered account.
- If you seed an email before that user signs up, the script will create a database user record without a password account, and that same email will no longer be able to register through the UI.

## Usage Instructions

### Local development

```bash
pnpm dev
```

Useful scripts:

| Command | Description |
| --- | --- |
| `pnpm dev` | Start the Next.js development server |
| `pnpm build` | Create a production build |
| `pnpm start` | Run the production server |
| `pnpm db:seed` | Seed sample data using `.env` |
| `pnpm db:seed:docker` | Seed sample data in containerized environments |
| `pnpm lint` | Run ESLint |
| `pnpm lint:fix` | Run ESLint with auto-fix |
| `pnpm type-check` | Run TypeScript checks |
| `pnpm format` | Format supported files with Prettier |
| `pnpm format:check` | Validate formatting |

### Everyday app flow

1. Sign up at `/signup`.
2. Verify your email.
3. Sign in at `/signin`.
4. Create tasks in the board lanes.
5. Edit or move only the tasks you own.
6. Watch the notification area for nearly expired and overdue tasks.
7. Open reminder links to jump directly to a task edit page.

### Docker Compose

Build and start the application and database:

```bash
docker compose up --build -d app db
```

Seed sample data:

```bash
docker compose run --rm seed
```

Stop the stack:

```bash
docker compose down
```

If you want a completely fresh Postgres init from the `drizzle` SQL files, remove the existing volume and rebuild the stack:

```bash
docker compose down -v
docker compose up --build
```

The Compose stack provides:

- `app`: production Next.js container
- `db`: PostgreSQL 16 with persisted data
- `seed`: one-off seeding container

## Production Notes

### Runtime behavior

- The production image uses a standalone Next.js build and runs with `node server.js`.
- The app exposes a health endpoint at `/api/health`.
- A Docker health check calls `/api/health` and also verifies database connectivity.

### Reminder monitor

- The reminder monitor starts from `instrumentation.ts` when the Node.js server boots.
- It checks tasks every `10` seconds.
- Reminder emails are guarded with PostgreSQL advisory locks to reduce duplicate sends across concurrent workers.

### Authentication and routing

- Non-API routes are protected by middleware in `proxy.ts`.
- Authenticated users are redirected away from `/signin` and `/signup`.
- Better Auth endpoints are mounted under `/api/auth/[...all]`.
- Sessions are configured for 7 days and use secure cookies in production.

### Deployment recommendations

- Run behind HTTPS in production.
- Use a strong `BETTER_AUTH_SECRET`.
- Use a managed PostgreSQL instance or persistent volume-backed Postgres.
- Provide a valid SMTP service before enabling user registration.
- Set `BASE_URL` and `NEXT_PUBLIC_BASE_URL` to the public production origin.

## Troubleshooting

### I can sign up but cannot sign in

Check SMTP delivery first. Email verification is required before sign-in succeeds.

### The app cannot connect to PostgreSQL

Verify:

- `DATABASE_URL` points to a reachable database
- `DATABASE_SSL` matches the target database requirements
- Your schema SQL in `drizzle/` has been applied

### Seeded tasks are missing

Verify that:

- You seeded the same email address as the account you sign in with
- The database already contains the schema
- You are not using different local and Docker database URLs by mistake
